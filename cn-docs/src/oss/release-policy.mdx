    ## 发布节奏

    :::python
    随着 LangChain 1.0 的发布，`langchain` 和 `langchain-core` 的**次要**版本发布（例如，从 `1.0.x` 到 `1.1.0`）遵循语义化版本控制，并且可能频繁发布。次要版本包含新功能和改进，但不包含破坏性变更。
    :::
    :::js
    随着 LangChain 1.0 的发布，`langchain` 和 `@langchain/core` 的**次要**版本发布（例如，从 `1.0.x` 到 `1.1.0`）遵循语义化版本控制，并且可能频繁发布。次要版本包含新功能和改进，但不包含破坏性变更。
    :::

    补丁版本会频繁发布，每周可能发布数次，因为它们包含错误修复和微小改进。

    ## API 稳定性

    :::python
    LLM 应用程序的开发是一个快速发展的领域，我们不断从用户和社区中学习。因此，我们预计 `langchain` 和 `langchain-core` 中的 API 将继续发展，以更好地满足用户的需求。
    :::
    :::js
    LLM 应用程序的开发是一个快速发展的领域，我们不断从用户和社区中学习。因此，我们预计 `langchain` 和 `@langchain/core` 中的 API 将继续发展，以更好地满足用户的需求。
    :::

    随着 LangChain 1.0 采用语义化版本控制：
    - 对公共 API 的破坏性变更只会在主版本发布时发生（例如，`2.0.0`）
    - 次要版本升级（例如，`1.0.0` 到 `1.1.0`）会添加新功能，而不引入破坏性变更
    - 补丁版本升级（例如，`1.0.0` 到 `1.0.1`）包含错误修复和微小改进

    我们通常会尽量避免进行不必要的更改，并为即将移除的功能提供弃用策略。

    ### 其他包的稳定性

    LangChain 生态系统中其他包的稳定性可能有所不同：

    - **由 LangChain 维护的合作伙伴包**（例如 `langchain-openai` 和 `langchain-anthropic`）遵循语义化版本控制，并预计在 1.0 版本后保持稳定。其他合作伙伴包可能遵循不同的稳定性和版本控制策略，用户应参考这些包的文档以获取更多信息。

    :::python
    - **`langchain-community`** 是一个社区维护的包，包含第三方集成。由于其中的集成数量众多，`langchain-community` 不遵循与 `langchain` 和 `langchain-core` 相同的严格语义化版本控制策略。更多详情请参阅长期支持下的“特殊注意事项”部分。
    :::
    :::js
    - **`@langchain/community`** 是一个社区维护的包，包含第三方集成。由于社区贡献和第三方依赖的性质，`@langchain/community` 不遵循与 `langchain` 和 `@langchain/core` 相同的严格语义化版本控制策略。更多详情请参阅长期支持下的“特殊注意事项”部分。
    :::

    ## 弃用策略

    我们通常会避免弃用功能，直到有更好的替代方案可用。

    :::python
    采用 LangChain 1.0 的语义化版本控制方法后，已弃用的功能在整个 1.x 发布系列中将继续有效。破坏性变更，包括移除已弃用的功能，只会在主版本发布时发生（例如，2.0）。

    当 `langchain` 或 `langchain-core` 中的某个功能被弃用时，我们将：
    - 在代码和文档中明确标记为已弃用
    - 提供迁移到推荐替代方案的指导
    - 在所有 1.x 次要版本中为已弃用的功能提供安全更新
    :::
    :::js
    采用 LangChain 1.0 的语义化版本控制方法后，已弃用的功能在整个 1.x 发布系列中将继续有效。破坏性变更，包括移除已弃用的功能，只会在主版本发布时发生（例如，2.0）。

    当 `langchain` 或 `@langchain/core` 中的某个功能被弃用时，我们将：
    - 在代码和文档中明确标记为已弃用
    - 提供迁移到推荐替代方案的指导
    - 在所有 1.x 次要版本中为已弃用的功能提供安全更新
    :::

    在某些情况下，如果已弃用的功能没有造成维护问题，我们可能会允许它们在代码库中保留更长时间，以进一步减轻用户的负担。

    ## 长期支持 (LTS)

    LangChain 遵循长期支持 (LTS) 策略，为生产应用程序提供稳定性：

    ### 发布状态定义

    包被标记为以下状态之一：
    - **活跃 (ACTIVE)**：当前处于积极开发阶段，包括错误修复、安全补丁和新功能
    - **维护 (MAINTENANCE)**：接收所有安全补丁和关键错误修复，但没有新功能

    ### 当前 LTS 版本

    **LangChain 1.0** 被指定为 LTS 版本：
    - **状态**：在 2.0 发布之前为活跃状态
    - **支持期限**：2.0 发布后，1.0 将进入维护模式至少 1 年
    - **语义化版本控制合规性**：用户可以在次要版本之间升级（例如，从 1.0 到 1.1）而不会遇到破坏性变更

    ### 旧版本支持

    **LangChain 0.3**：
    - **状态**：维护模式
    - **支持期限**：直到 2026 年 12 月
    - **支持内容**：安全补丁和关键错误修复

    ### 特殊注意事项

    :::python
    **langchain-community 0.4**：由于社区贡献和第三方集成的性质，`langchain-community` 可能在次要版本发布时包含破坏性变更。它已发布为 0.4 版本，以反映这种不同的稳定性策略。
    :::
    :::js
    **@langchain/community 1.0**：由于社区贡献和第三方集成的性质，`@langchain/community` 可能在次要版本发布时包含破坏性变更。
    :::

    <Tab title="LangGraph">
    LangGraph 遵循结构化的发布策略，以确保为构建生产应用程序的用户提供稳定性和可预测性。

    ## 发布节奏

    我们预计**主**版本发布间隔至少为 6-12 个月，以为生产应用程序提供稳定性。

    **次要**版本通常每 1-2 个月发布一次，包含新功能和改进。

    **补丁**版本根据需要发布，通常是每周，以解决错误和安全问题。

    ## API 稳定性

    ### 稳定 API

    所有没有特殊前缀的 API 都被认为是稳定的，可以用于生产。我们在一个主版本内为稳定功能保持向后兼容性。

    ### Beta 功能

    在文档中标记为 `beta` 的功能具有以下特点：
    - 功能完整且经过测试
    - 可用于生产，但需理解它们可能会发生变化
    - 可能根据用户反馈进行小幅 API 调整

    ### 实验性功能

    标记为 `experimental` 或 `alpha` 的功能：
    - 处于积极开发阶段
    - 可能发生重大变化或被移除
    - 在生产中应谨慎使用

    ### 内部 API

    以下划线 (`_`) 为前缀或明确标记为内部的 API：
    - 不属于公共 API
    - 可能在没有通知的情况下更改
    - 不应直接使用

    ## 弃用策略

    弃用功能时：
    1.  **弃用通知**：功能被标记为已弃用，并提供清晰的迁移指导
    2.  **宽限期**：已弃用的功能至少在下一个次要版本中保持功能正常
    3.  **移除**：功能仅在主版本发布时被移除
    4.  **迁移支持**：我们提供迁移指南，并在可能的情况下提供自动化工具

    ## 平台兼容性

    :::python
    ### Python 支持

    - 我们支持 Python 软件基金会积极维护的 Python 版本
    - Python 版本要求可能只在主版本发布时更改
    - 当前要求 Python 3.10 或更高版本
    :::

    :::js
    ### JavaScript/TypeScript 支持

    - 我们支持 Node.js LTS 版本
    - 为所有公共 API 提供 TypeScript 定义
    - 客户端组件的浏览器兼容性已记录在案
    :::

    ## 破坏性变更

    破坏性变更仅在主版本中引入，包括：
    - 移除已弃用的 API
    - 更改必需参数
    - 影响现有应用程序的默认行为变更
    - 最低 Python/Node.js 版本更新

    ## 迁移支持

    对于主版本升级，我们提供：
    - 全面的迁移指南
    - 在可行的情况下提供自动化迁移脚本
    - 对前一个主版本的延长支持期
    - 所有破坏性变更的清晰文档
